### 8-1. "명령형 리더십"에서 "실험형 리더십"으로

"자, 피닉스 2.0 시작합니다."

김철수 실장이 말했다. 하지만 분위기가 달랐다.

예전 같았으면 50페이지짜리 기획서를 화면에 띄우고 "이대로 하면 됩니다"라고 했을 것이다. 하지만 오늘은 화면이 비어 있었다.

"이번에는 제가 다 정하지 않을 거예요."

"네?"

"여러분이 정해보세요."

팀원들이 서로 눈을 마주쳤다.

"ㅋㅋ 괜찮아요. 왜?"

"갑자기 민주주의시라서요."

"네. 내가 다 정했더니 망했잖아요. 그래서 이번엔 다르게 해보려고요."

### 8-2. 기술 스택 전쟁: 논쟁에서 합의까지

"저는 TypeScript + React + Next.js를 제안합니다."

박지은이 화면을 공유했다.

```
📊 기술 스택 제안서

제안: TypeScript + React + Next.js

장점:
- 타입 안정성 (런타임 에러 감소)
- 풍부한 생태계 (npm 패키지)
- SSR 지원 (SEO, 성능)
- 커뮤니티 활성화 (문제 해결 용이)

단점:
- 러닝 커브 (TypeScript)
- 빌드 시간 증가
```

김철수 실장이 고개를 끄덕였다.

"배우는 시간은 일정에 포함됐어?"

"..."

"안 했지?"

"... 네."

"그거 넣어줘. 1주일 정도?"

"네. 추가할게요."

예전의 김철수 실장이었다면 "그것도 몰라?"라고 했을 것이다. 하지만 지금은 달랐다.

"MVP라서요. 빨리 만들고 빨리 테스트하는 게 중요한데, Spring은 초기 세팅이 오래 걸려요."

김철수 실장이 생각했다.

"... 그것도 맞네."

```
📊 Node.js vs Spring 비교 (이번 프로젝트 기준)

| 항목 | Node.js | Spring |
|-----|---------|--------|
| 초기 세팅 | 2시간 | 1-2일 |

| 배포 복잡도 | 낮음 | 중간 |
| 성능 | 충분함 | 오버스펙 |

결론: 이번 MVP에는 Node.js가 적합
```

김철수 실장이 데이터를 보고 고개를 끄덕였다.

"알겠어. 데이터로 설득하니까 납득이 되네."

"진짜요?"

"응. Node.js로 하자."

박지은이 깜짝 놀랐다.

"실장님, 진짜 바꾸실 거예요?"

"데이터가 저렇게 나왔는데 뭘. 내 고집보다 프로젝트 성공이 중요하지."

**페어 프로그래밍 1일 차**

"내가 먼저 할게."

이준호 선임이 키보드를 잡았다.

"여기서 API 호출하고..."

"잠깐요. 거기 async/await 쓰면 안 돼요?"

"응? 난 Promise 체인이 익숙해서..."

"그거 콜백 지옥이랑 비슷해요. async/await이 가독성이 좋아요."

이준호 선임이 멈췄다.

"그래? 해봐."

박지은이 키보드를 받았다.

```typescript
// Before (이준호 방식)
getUser()
  .then(user => getOrders(user.id))
  .then(orders => processOrders(orders))
  .catch(err => handleError(err));

// After (박지은 방식)
async function fetchUserOrders() {
  try {
    const user = await getUser();
    const orders = await getOrders(user.id);
    return processOrders(orders);
  } catch (err) {
    handleError(err);
  }
}
```

"오, 확실히 읽기 편하네."

"그렇죠?"

"근데 에러 처리는 이렇게 하면 더 나을 것 같은데..."

두 사람의 논쟁이 시작됐다. 하지만 싸움이 아니라 토론이었다.

"ㅋㅋ 벌금이요."

"ㅋㅋ 알아. 이번 달 벌금 3만 원 쌓였어."

변화는 계속되고 있었다. 완벽하지 않았지만, 확실히 나아지고 있었다.

한 달 약속의 반이 지났다. 그리고 팀은 이미 달라져 있었다.
