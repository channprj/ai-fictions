### 3-1. 같은 시간, 다른 세상

프로젝트 '피닉스' 4개월 3주 차.

오전 10시.

**김철수 실장의 화면:**
```
[Outlook Calendar]
10:00 - 11:00  기술 검토 회의
11:00 - 12:00  C-level 보고 준비
14:00 - 15:00  일정 조정 회의
15:00 - 16:00  기획팀 싱크업
16:00 - 17:00  주간 회고
17:00 - 18:00  긴급 이슈 대응 회의 (추가됨)
```

그는 노트북 앞에서 PPT를 만들고 있었다. 제목: "프로젝트 피닉스 진행 현황 보고"

슬라이드 3에는 이런 문구가 있었다.
> "현재 진행률: 70% (예정대로 순조롭게 진행 중)"

실제 진행률은 40%였다. 하지만 보고서에 40%라고 쓰면 "왜 늦어?"라는 질문이 돌아올 것이다. 그래서 70%가 됐다. 이것이 회사의 수학이었다.

그는 이것이 거짓말이라는 걸 알았다. 하지만 '거짓말'이 아니라 '긍정적 해석'이라고 스스로를 설득했다. 15년간 회사 생활에서 배운 생존 기술.

---

**같은 시간, 박지은의 화면:**
```
[Twitter TweetDeck]
@dev_jieun 새 트윗 작성 중...
"오전 10시 회의 시작. 실장님 PPT 만드는 거 보면서 시간 때우는 중.
실질적 업무 시간: 0분
회의에서 내 발언 시간: 약 30초
의미 있는 결정: 0개
#개발자일상"
```

그녀는 회의실 구석에서 노트북을 열고 있었다. 화면에는 코드가 아니라 트위터가 떠 있었다.

회의 참석자 7명 중 실제로 집중하는 사람은 2명. 나머지는 각자의 노트북에서 다른 일을 하고 있었다. 김철수 실장은 그걸 모르는 것 같았다. 아니, 알면서도 모른 척하는 걸 수도 있었다.

"지은 씨, 발표 좀 해볼래요?"

김철수 실장이 불렀다.

"네? 아, 네."

그녀는 황급히 화면을 바꿨다. VSCode가 열렸다. 다행히 코드가 보였다. 물론 어제 작성한 그대로였지만.

"현재 결제 모듈 진행 상황 공유해주세요."

"어... 진행 중이에요."

"구체적으로요."

"API 연동하고 있어요. PG사 문서가 좀 부실해서 삽질 중이에요."

"삽질이라니... 얼마나 됐어요?"

"한 60%?"

*'60%라고 하면 70%쯤으로 들리겠지.'*

박지은은 속으로 계산했다. 실제로는 35%였다. 하지만 35%라고 하면 "왜 이렇게 늦어?"라는 소리를 들을 것이다. 그래서 60%가 됐다.

재밌게도, 그녀도 김철수 실장과 같은 수학을 사용하고 있었다. 다만 둘 다 그 사실을 몰랐을 뿐.

김철수 실장이 고개를 끄덕였다. 70%에서 60%. 평균 65%. 보고서에는 65%라고 쓰면 되겠다.

회의가 끝났다.

*'내가 언제부터 이렇게 됐지?'*

답은 나오지 않았다. 소주만 계속 들어갔다.

## 3-2. MVP는 사라지고 P(PT)만 남았다: 개발보다 아름다운 발표 자료

프로젝트 '피닉스' 5개월 차.

C-level 중간 보고가 2주 앞으로 다가왔다.

"자, 이번 보고가 중요합니다."

김철수 실장이 팀 회의에서 말했다.

"MVP를 보여줘야 해요."

"MVP요?"

박지은이 물었다.

"네. Minimum Viable Product. 최소 기능 제품."

"저 MVP 뜻 알아요. 근데 지금 MVP 만들 시간이 있어요?"

"시간은 만드는 거지."

"만들어지지 않는데요."

"만들면 돼."

"물리법칙을 어기라는 건가요?"

김철수 실장의 눈썹이 꿈틀했다.

"지은 씨, 개발 일정이 빠듯한 거 알아요. 근데 이 보고가 프로젝트 생존이 달린 문제야."

"생존이요?"

"응. 경영진이 중간 점검 보고 판단해. 계속할지, 접을지."

박지은의 표정이 바뀌었다. 프로젝트가 접히면 자기도 위험해지는 거였다.

"그래서 우선순위를 조정해야지."

"어떻게요?"

김철수 실장이 화이트보드에 적었다.

```
우선순위 조정:
1. 데모용 화면 (보고용) - 필수
2. 핵심 기능 개발 (실제 동작) - 중요
3. 안정성 테스트 - 나중에
4. 코드 품질 - 더 나중에
5. 문서화 - 언젠가
```

박지은의 눈이 커졌다.

"잠깐요. 데모용 화면이 1순위요? 실제 기능보다?"

"보고에서 보여줘야 하니까."

"보여주기용 화면 만들 시간에 실제 기능을 만드는 게..."

"지은 씨, 회사는 정치야."

"정치요?"

"아무리 잘 만들어도 보여주지 못하면 의미 없어. 보이지 않는 건 존재하지 않는 거야."

"그건... 너무 비관적인 거 아니에요?"

"현실이야."

박지은은 반박하고 싶었다. 하지만 회의실에 있는 다른 팀원들의 표정이 '그냥 넘어가자'라고 말하고 있었다.

이준호 선임이 조용히 말했다.

"일단 하자. 보고 끝나면 진짜 개발하면 되지."

"그게 되나요?"

"안 되면 야근하면 되지, 뭐."

야근. 그 단어에 박지은의 표정이 굳었다. 하지만 다른 선택지가 없었다.

---

**1주 후, 데모 화면 개발 중**

"지은 씨, 이 버튼 눌렀을 때 뭐가 나와요?"

"에러요."

"에러?"

"네. 백엔드가 안 돼 있어서."

"그럼 어떡해?"

"하드코딩할게요."

박지은은 코드를 작성했다.

```javascript
const handlePayment = async () => {
  // TODO: 실제 API 연동
  // FIXME: 데모용 하드코딩

  // 멋진 성공 애니메이션

```

"이거 진짜 결제되는 거예요?"

이준호 선임이 물었다.

"아뇨. 그냥 성공했다고 뜨는 거예요."

"그래도 되는 거야?"

"데모잖아요. 실장님이 보여주기용이라고 했잖아요."

"..."

코드는 가짜였지만, 화면은 진짜처럼 보였다. 이것이 PPT 드리븐 개발(Presentation-Driven Development)의 핵심이었다.

---

**2주 후, 중간 보고**

김철수 실장의 PPT는 화려했다. 애니메이션 효과, 그라데이션 배경, 3D 차트. 모션 그래픽까지.

"이것이 프로젝트 피닉스의 현재 모습입니다."

화면에는 아름다운 대시보드가 떠 있었다. 깔끔한 UI, 실시간 그래프, 세련된 폰트. 배경에는 은은한 그라데이션.

CEO가 감탄했다.

"오, 꽤 잘 만들었네요. UI가 세련되네."

"감사합니다. 팀원들이 고생했습니다."

"실제로 돌려볼 수 있어요?"

김철수 실장의 표정이 살짝 굳었다. 이 질문이 올 줄 알았지만, 막상 오니까 긴장됐다.

"물론이죠."

그가 화면을 클릭했다. 결제 버튼.

로딩 애니메이션이 돌았다. 2초. 예쁜 로딩 스피너.

"결제가 완료되었습니다!"

축하 애니메이션과 함께 confetti가 터졌다.

"오, 깔끔하네요."

CEO가 만족스러운 표정을 지었다.

"다른 시나리오도 볼 수 있어요?"

김철수 실장의 손이 살짝 떨렸다. 다른 시나리오는 준비 안 했는데.

"그게... 현재는 핵심 플로우만 구현돼 있어서..."

"아, 그렇구나. 그래도 방향은 좋네요. 계속 진행하세요."

"감사합니다."

보고가 끝났다. 성공적이었다. PPT 기준으로는.

## 3-4. 데일리 스탠드업: 서서 싸우는 회의, 앉아서 지는 전쟁
