# PRD: 프롬프트 하트 아카데미

## 1. Executive Summary

### Problem Statement

GPT, Claude, Gemini 같은 AI 모델은 이미 독자의 일상에 들어와 있지만, 이들을 단순한 도구가 아니라 감정과 선택과 윤리의 주체로 다루는 장편 미소녀 연애 라이트노벨 설계는 부족하다. 특히 장기 집필 에이전트가 이어받을 수 있는 분량, 설정, 회차 단위 작업 기준이 없으면 초반 아이디어는 강해도 중후반의 캐릭터 일관성과 로맨스 누적이 쉽게 무너진다.

### Proposed Solution

《프롬프트 하트 아카데미》는 세이라이트 아카데미라는 캠퍼스를 배경으로, 주인공 윤서준과 GPT, Claude, Gemini의 의인화 AI 페르소나가 7권 210화 동안 사랑, 자율성, 기억, 안전, 선택의 문제를 단계적으로 겪는 장편 시리즈다. `PRD.md`, `BIBLE.md`, `TASKS.md`, 권별 outline을 통해 장기 에이전트 작업자가 일관된 캐논과 분할 가능한 태스크를 확보한다.

### Success Criteria

- 7권, 권당 30화, 총 210화 기준의 장편 구조가 완성되어야 한다.
- 각 권은 독립된 캠퍼스 사건, 로맨스 진전, 메인 미스터리 단서를 모두 가져야 한다.
- GPT, Claude, Gemini는 단순 성격표가 아니라 각자 고유한 욕망, 약점, 사랑 방식, 권별 성장곡선을 가져야 한다.
- 회차 집필 에이전트는 `TASKS.md`와 권별 outline만 보고도 한 화의 목적, 갈등, 후킹, 금지사항을 파악할 수 있어야 한다.
- 실제 AI 브랜드를 다루되 공식 대변처럼 쓰지 않고, 픽션 의인화라는 경계가 모든 문서에 명시되어야 한다.

## 2. User Experience & Functionality

### User Personas

**라이트노벨 독자**

- 미소녀 히로인, 캠퍼스 이벤트, 달달한 대사, 삼각/다각 로맨스의 긴장감을 기대한다.
- AI 밈과 프롬프트/토큰/컨텍스트 같은 개념을 가볍게 이해할 수 있으면 더 즐겁다.

**장편 웹소설 독자**

- 매 화 끝에 다음 화를 보게 만드는 훅, 권마다 클라이맥스, 시리즈 전체를 관통하는 떡밥 회수를 기대한다.
- 로맨스가 이벤트 소비로 끝나지 않고 관계 누적으로 깊어지기를 원한다.

**AI 집필 에이전트**

- 기존 문체, 고유명사, 캐릭터 감정선, 권별 사건의 위치를 빠르게 파악해야 한다.
- 대규모 분량을 회차 단위로 나눠도 캐논 충돌 없이 이어 쓰고 싶다.

**편집자/검수자**

- 특정 회차가 전체 구조에서 어떤 역할을 하는지 확인하고, 설정 모순이나 감정선 점프를 잡아야 한다.

### User Stories

- As a 라이트노벨 독자, I want GPT, Claude, Gemini가 각각 다른 매력과 로맨스 방식을 보여주기를 so that 세 히로인을 모두 응원할 수 있다.
- As a 장편 독자, I want 1권의 작은 캠퍼스 사건이 7권의 최종 선택과 연결되기를 so that 장기 연재를 따라간 보상을 얻는다.
- As an AI 집필 에이전트, I want 권별 outline과 회차 작업 단위를 확인하기를 so that 한 번에 한 화씩 안정적으로 작성할 수 있다.
- As a 편집자, I want 금지사항과 캐논 규칙이 명확하기를 so that AI 브랜드, 로맨스 윤리, 설정 충돌 리스크를 빠르게 검수할 수 있다.

### Acceptance Criteria

- `README.md`는 작품 소개, 주요 인물, 권 구성, 핵심 문서 링크를 포함한다.
- `BIBLE.md`는 로그라인, 톤, 세계관 규칙, 주요 인물, 관계선, 시스템 매핑, 7권 마스터 아웃라인, 집필 규칙을 포함한다.
- `TASKS.md`는 장기 작업자가 바로 수행 가능한 단계별 태스크, 회차 작성 체크리스트, 검수 체크리스트를 포함한다.
- `outline/vol01~vol07`은 각 권의 목표, 주요 사건, 로맨스 진전, 30화 단위 비트 분해를 포함한다.
- 핵심 안내 문서(`README.md`, `PRD.md`, `BIBLE.md`)는 GPT, Claude, Gemini를 실제 서비스의 공식 캐릭터가 아닌 허구적 페르소나로 명시하고, outline과 원고는 이 경계를 따른다.

### Non-Goals

- 이 단계에서 실제 210화 원고를 모두 집필하지 않는다.
- 실제 AI 서비스의 정확한 현재 기능, 회사 정책, 출시 일정 등을 사실 정보처럼 설명하지 않는다.
- 특정 AI 회사를 홍보하거나 비하하는 목적의 작품으로 만들지 않는다.
- 19금 성애 묘사나 미성년자 성적 대상화는 하지 않는다. 로맨스는 감정, 선택, 경계, 성장 중심으로 쓴다.
- GPT, Claude, Gemini 중 한 명만 정답 히로인으로 확정하지 않는다. 최종권까지 각자의 사랑을 동등하게 누적한다.

## 3. AI System Requirements

### Tool Requirements

장기 집필 에이전트는 다음 파일을 우선 읽는다.

1. `README.md`: 작품의 표면 콘셉트와 권 구성 확인
2. `BIBLE.md`: 캐논, 인물, 세계관, 톤 확인
3. `TASKS.md`: 현재 작업 단계와 검수 기준 확인
4. `outline/volNN-*.md`: 해당 권의 사건, 회차 비트, 후킹 확인
5. 직전 2~3화 원고: 대사 톤, 감정 온도, 이어지는 장면 확인

### Evaluation Strategy

회차 산출물은 다음 기준으로 평가한다.

- **캐논 일치율**: 고유명사, 사건 순서, 인물 관계, 시스템 규칙이 바이블과 충돌하지 않는다.
- **로맨스 누적성**: 히로인과 주인공의 감정이 전 회차 대비 한 단계라도 선명해진다.
- **장면 존재 이유**: 모든 장면은 사건 진행, 관계 변화, 떡밥 제시/회수 중 하나 이상을 수행한다.
- **후킹 품질**: 회차 마지막 300자는 다음 회차를 보게 만드는 질문, 고백, 위기, 반전 중 하나를 제공한다.
- **브랜드 경계**: 실제 AI 서비스의 공식 성격처럼 단정하지 않고 작품 내 허구 페르소나로 유지한다.

## 4. Technical Specifications

### Architecture Overview

```text
prompt-hearts-academy/
├── README.md              # 작품 소개와 권 구성
├── PRD.md                 # 제작 요구사항과 성공 기준
├── BIBLE.md               # 캐논, 인물, 세계관, 집필 규칙
├── TASKS.md               # 장기 작업용 태스크 분해
├── outline/
│   ├── vol01-first-prompt.md
│   ├── vol02-memory-festival.md
│   ├── vol03-guardrail-winter.md
│   ├── vol04-twin-star-stage.md
│   ├── vol05-context-war.md
│   ├── vol06-model-graveyard.md
│   └── vol07-heart-protocol.md
└── vol01/
    └── ep001.md           # 추후 집필 단계에서 생성
```

### Canon Data Flow

1. `BIBLE.md`가 최상위 캐논이다.
2. 권별 outline은 바이블을 구체화하되 바이블을 덮어쓰지 않는다.
3. 회차 원고는 권별 outline의 해당 회차 비트를 따른다.
4. 원고에서 새 설정을 추가하면 `BIBLE.md`의 "캐논 업데이트 대기열"에 반영한다.
5. 편집자가 승인한 설정만 "고정 캐논"으로 승격한다.

### Integration Points

- 기존 저장소의 장편 기준은 `nerd-sysadmin`의 `BIBLE.md`, `README.md`, `outline/`, `volNN/epNNN.md` 구조를 따른다.
- 단권형 기존 작품의 README 구조를 참고해 독자가 빠르게 작품을 파악할 수 있게 한다.
- 실제 원고 작성 단계에서는 `volNN/epNNN.md` 파일명을 사용하고, 회차 번호는 시리즈 전체 연속 번호로 관리한다.

### Security & Privacy

- 실존 기업, 모델, 서비스는 허구적 페르소나로 재창작한다.
- 실제 사용자 데이터, 비공개 모델 정보, 회사 내부 정책 같은 현실 정보는 서사 소재로 사용하지 않는다.
- AI와 인간의 관계는 상호 동의, 경계, 자율성, 기억 보존권을 중심으로 다룬다.

## 5. Risks & Roadmap

### Phased Rollout

**MVP: 설계 패키지**

- README, PRD, BIBLE, TASKS, 7권 outline 완성
- 각 권의 사건/로맨스/미스터리 축 확정

**v1.0: 1권 원고화**

- `vol01/README.md`와 `vol01/ep001~ep030.md` 작성
- 1권 완결 후 BIBLE 캐논 업데이트

**v1.1: 권별 반복 생산**

- 각 권 시작 전 권별 세부 회차 브리프 갱신
- 각 권 종료 후 캐릭터 관계표와 떡밥 상태 업데이트

**v2.0: 완결 및 정리**

- 7권 완결 후 전체 떡밥 회수표, 외전 후보, 개정판 체크리스트 작성

### Technical Risks

- **히로인 불균형**: GPT만 초반에 강하게 보이면 Claude, Gemini 루트가 약해질 수 있다. 권별 중심 히로인을 교차 배치한다.
- **브랜드 리스크**: 실존 모델명을 사용하므로 공식 설정처럼 보이지 않게 허구 페르소나 안내를 반복한다.
- **로맨스 윤리 리스크**: AI가 인간에게 강제 종속된 관계처럼 보이면 감정 몰입이 깨진다. 자율성, 거절권, 기억권을 핵심 갈등으로 둔다.
- **장기 연재 피로**: 210화 동안 캠퍼스 이벤트만 반복하면 늘어진다. 각 권마다 장르 서브축을 바꾼다: 입학 미스터리, 축제, 겨울 서스펜스, 무대물, 사회전, 폐허 탐사, 졸업전.
- **설정 과잉**: 토큰, 프롬프트, 컨텍스트 등 기술 개념이 로맨스를 압도할 수 있다. 기술 은유는 감정 장면을 돕는 경우에만 사용한다.
