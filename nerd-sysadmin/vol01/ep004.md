# 제 4화: 레거시 코드 리뷰

[← 이전화: 제 3화](../vol01/ep003.md) | [목차](./README.md) | [다음화: 제 5화 →](../vol01/ep005.md)


`fire/`. 화염 마법 디렉토리.

세계의 마법이 폴더별로 정갈하게 나뉘어 있는 걸 보고 나는 잠깐, 아주 잠깐 안심했었다. `fire/` `water/` `earth/` `wind/` `light/` `dark/`. 원소별로 디렉토리 구조가 잡혀 있다는 건, 적어도 이 세계를 짠 놈이 객체지향이라는 단어 정도는 들어봤다는 뜻이니까.

순진한 생각이었다.

폴더 구조가 깔끔하다고 코드가 깔끔하리란 보장은 없다. 오히려 정반대인 경우가 더 많다. 디렉토리만 그럴듯하게 발라놓고 그 안의 함수는 죄다 똥인 프로젝트를, 나는 전생에서 수백 개쯤 봤다. 겉만 번지르르한 README, 열어보면 지옥. 흔히들 'README 드리븐 디벨롭먼트'라고 부르는 그것.

나는 침대 위에 책상다리를 하고 앉아, 허공에 떠 있는 검은 터미널 창에 손가락을 가져갔다.

```
root@lumen-territory:~# cat /sys/magic/fire/ignite.mag
```

엔터.

화면 가득 코드가 출력되는 순간, 나는 숨을 멈췄다.

그리고 5초 뒤, 길게 내뱉었다.

```
function cast_ignite() {
    allocate_mana(100);
    wait(3000); // Gather energy from nature

    for (int i = 0; i < 10; i++) {
        check_temperature(i);
        increase_heat(5);
    }

    if (mana >= 100) {
        release_fire();
    } else {
        fail_magic();
    }
}
```

"......이게."

나는 한 줄, 한 줄, 천천히 눈으로 훑었다. 혹시 내가 잘못 본 게 아닐까 싶어서. 아르시안의 뇌에 남아 있는 희미한 지식이 친절하게 속삭여줬다. 이게 전 대륙의 마법사들이 수십 년간 외우고 다니는, 그 유명한 '파이어볼'의 기본 원리라고.

"이게 파이어볼의 기본 원리라고?"

헛웃음이 나왔다.

전생의 나였다면 지금 당장 사내 코드 리뷰 채널에 이 스니펫을 박제하고 멘션을 갈겼을 거다. *"이거 누가 짰어요? 자수하세요. 안 잡아먹어요. 그냥 옆자리로 잠깐 오시면 됩니다."* 그리고 그 자리는, 장담하건대, 옆자리가 아니었을 거다. 정신과 상담실 정도 거리는 됐겠지.

자, 코드 리뷰를 시작하자. 전생 8년차 짬밥으로, 한 줄씩 발라주마.

**1번. `allocate_mana(100);`**

함수 첫 줄부터 마나를 100이나 선점한다. 무려 100. 화염구 하나 만드는 데 이 세계 마나 단위로 100이 잡힌다는 소리다. 근데 코드 어디에도 이 100을 다 쓰는 구간이 없다. 실제로 불을 뿜는 `release_fire()`는 인자도 안 받는다. 그러니까 100을 잡아놓고, 정작 얼마를 쓰는지는 아무도 모른다는 거다. 이건 마법이 아니라 메모리 선점 깡패다. 서버로 치면 요청 하나 처리하려고 힙(Heap)을 통째로 예약 거는 꼴.

**2번. 그리고 이 새끼.** `wait(3000);`

```
    wait(3000); // Gather energy from nature
```

나는 이 한 줄을 한참 노려봤다.

`wait(3000)`. 밀리초 단위라면 3초. 주석은 친절하게도 *"Gather energy from nature(자연에서 기운을 모은다)"*라고 적혀 있다.

자연에서 기운을 모은다고? 개소리하고 있네.

이건 그냥 **`sleep`**이다. 아무것도 안 하고 3초간 스레드를 멈춰 세우는, 순수하고 정직한 블로킹(Blocking)이다. 자연이고 나발이고, 시스템은 이 3초 동안 단 한 톨의 연산도 하지 않는다. 그냥 멍 때리고 서 있는다.

순간 머릿속에서 퍼즐 조각이 맞춰지는 소리가 났다.

이세계 마법사들이 불꽃 하나 만드는 데 왜 그렇게 거창하게 영창을 웅얼거리는지. 눈을 감고, 두 손을 모으고, "대지의 정령이여, 태초의 불꽃이여, 내게 화염의 권능을……" 어쩌고 하면서 몇 분씩 폼을 잡는지.

그게 마법사의 정신 집중이라고? 영적 교감이라고?

아니. 그냥 시스템이 `wait(3000)` 돌리는 동안 어색하니까 입을 터는 거였다. 컴파일 돌아가는 3분 동안 의자에 기대 커피 마시는 개발자랑 똑같은 거였다. 영창은 정신 집중이 아니라 **로딩 화면**이었던 거다. 마법사들의 그 장엄한 주문이, 알고 보니 프로그레스 바 채워지는 거 멍하니 보는 행위였다.

"......이걸 수백 년을 그랬다고?"

전 대륙의 마법사들이, 단지 누군가 `sleep`을 하드코딩으로 박아놨다는 이유 하나로, 수백 년간 화염구 하나 쏠 때마다 3초씩 손해를 보고 있었다. 인류 문명 단위의 시간 낭비. 누적하면 대체 몇 명의 인생이 `wait` 안에서 증발한 걸까.

**3번. for 루프.**

```
    for (int i = 0; i < 10; i++) {
        check_temperature(i);
        increase_heat(5);
    }
```

온도를 올리는 구간. 이건 그나마 일을 하긴 한다. 문제는 *하는 방식*이다.

`increase_heat(5)`를 열 번 돌린다. 한 번에 5씩, 열 번. 그럼 그냥 `increase_heat(50)` 한 번 부르면 되잖아? 굳이 루프를 열 번 돌면서, 매 회마다 `check_temperature(i)`로 온도까지 측정한다. 측정한 값으로 뭘 하느냐? 아무것도 안 한다. 분기도 없고, 조건도 없다. 그냥 잰다. 재고 버린다. 재고 버린다. 열 번.

이건 온도계를 들고 냄비를 열 번 들여다보면서, 정작 불은 균일하게 5씩만 올리는 행위다. 측정 결과를 아무 데도 안 쓸 거면 측정을 왜 해. 마나 회로 자원만 갉아먹는 순수 오버헤드. 이 세계 기준으로는 CPU 사이클이 아니라 마나 사이클을 태우고 있는 거다.

**4번. 그리고 대망의 `if`.**

```
    if (mana >= 100) {
        release_fire();
    } else {
        fail_magic();
    }
```

함수 맨 위에서 `allocate_mana(100)`으로 100을 잡았다. 그리고 맨 아래에서 `mana >= 100`인지 검사한다.

방금 100을 할당해놓고, 이제 와서 100 이상인지 묻는다. 자기가 방금 채워놓은 통장 잔고를, 채워넣자마자 "잔액 충분하십니까?"라고 묻는 ATM이다. 당연히 충분하지. 네가 방금 넣었잖아. 이 `else` 분기, 즉 `fail_magic()`은 사실상 **데드 코드(Dead Code)**다. 영원히 도달하지 못하는 길. 코드 커버리지 측정기 돌리면 빨간 줄 쫙 그어질, 그 슬픈 라인.

리뷰 끝.

총평: 마나는 과잉 선점, 3초는 순수 낭비, 루프는 무의미, 분기는 죽은 코드. 이 짧은 함수 하나에 안티패턴을 전부 모아놨다. 무슨 안티패턴 종합선물세트도 아니고.

"어떤 미친놈이…… 코드를 이따위로 짠 거야?"

나는 진심으로 궁금했다. 이 세계의 창조자라는 작자는, 대체 어떤 정신 상태로 화염 마법 하나를 이 모양으로 만들어놓고 우주 배포를 갈겨버린 걸까. 스테이징도 안 거치고? QA도 없이? 이걸 그대로 프로덕션에? 게다가 롤백도 안 되는 단일 우주에?

전생에 무중단 배포 한 번 잘못 눌렀다가 새벽 3시에 전화받고 식은땀 흘리던 내 입장에서는, 이건 거의 신성모독 수준의 무책임이었다. 신이라며. 전지전능하다며. 근데 코드 리뷰는 안 받았네.

손가락이 근질거렸다.

이건 거의 생리적인 반응이었다. 전생에서, 더러운 스파게티 코드를 보면 참지 못하고 야근을 자처해가며 밤새 뜯어고치던 그 직업병. 퇴근 시간 지나서까지 남의 레거시를 리팩토링하다가, 결국 그 과로로 심장이 멎어 여기까지 굴러떨어진 그 빌어먹을 직업병이.

근데 어쩌겠나. 손이 먼저 움직이는걸.

```
root@lumen-territory:~# vi /sys/magic/fire/ignite.mag
```

가상의 터미널에서 vi 에디터가 열렸다. 동시에 시뻘건 경고문이 떴다.

```
[WARNING] Read-Only file system.
[WARNING] You need Root privileges to modify core magic logic.
```

리드 온리. 세계의 핵심 로직은 함부로 건드리지 못하게 잠가놨다는 소리다. 하긴, 이 정도 분별은 있었구나. 못 짠 건 못 짠 거고, 그래도 코어를 락 걸어둘 줄은 알았네. 칭찬해줄 뻔했다.

문제는, 그 락이 나한테는 아무 의미가 없다는 거다.

"이미 루트 권한은 내 손에 있다고."

`sudo`. 신께서 잠가둔 문을, 신께서 발급해주신 만능 열쇠로 연다. 이 아이러니를 신은 알고 있을까. 경고문이 스르륵 사라지고, 커서가 깜빡이기 시작했다.

자, 그럼.

나는 `dd`로 함수 전체를 통째로 날려버렸다. 미련 없이. 레거시를 리팩토링할 때 가장 짜릿한 순간은, 사실 새 코드를 짜는 게 아니다. 낡은 똥덩어리를 한 방에 비워버리는 그 순간이다. 빈 화면. 백지. 가능성. 개발자의 카타르시스.

그리고 다시 채워넣기 시작했다.

먼저 `wait(3000)`. 이 빌어먹을 3초. 가장 먼저, 가장 통쾌하게 지웠다. 자연에서 기운을 모은다고? 자연은 됐고, 나는 비동기로 모은다. 캐스팅하는 동안 멍 때리며 스레드 멈춰 세우는 짓 따위, 21세기 백엔드 개발자에게는 멸종한 패러다임이다. `async`/`await`. 무영창. 즉시 시전. 이게 현대다.

마나 할당량 `100`은 `10`으로 깎았다. 어차피 쓰지도 않을 90을 선점해서 뭐 하나. 필요한 만큼만 잡는다. 리소스 효율의 기본 of 기본.

의미 없는 for 루프는 통째로 들어냈다. 온도를 단계별로 찔끔찔끔 올릴 이유가 없다. 어차피 목표 온도는 정해져 있으니, 한 번의 호출로 임계점까지 즉시 가열한다. 측정? 안 한다. 측정해서 안 쓸 거면 안 재는 게 맞다.

데드 코드인 `else { fail_magic(); }`? 당연히 삭제. 도달하지 못하는 길은 길이 아니다.

남은 건 단 세 줄이었다.

```
async function cast_ignite_v2() {
    allocate_mana(10);
    await instant_ignition(1000); // 1000도까지 즉시 가열
    release_fire();
}
```

나는 화면을 바라보며 만족스럽게 고개를 끄덕였다.

`allocate_mana(10)` — 필요한 만큼만 정직하게 잡는다.
`await instant_ignition(1000)` — `wait(3000)` 같은 블로킹은 없다. 비동기로, 1000도까지, 즉시. 영창? 그게 뭐죠, 먹는 건가요.
`release_fire()` — 점화. 끝.

13줄짜리 스파게티를 3줄짜리 클린 코드로. 마나 소모는 10분의 1, 캐스팅 시간은 0초. 코드 라인 수는 줄었는데 화력은 오를 거다. 줄어든 라인 수만큼 늘어나는 효율. 이게 리팩토링의 미학이다. 덜어내는 게 더하는 거다.

"좋아. 코드가 훨씬 깔끔해졌군."

나는 콧노래라도 흥얼거리고 싶은 기분이었다. 8년간 굳은 직업병이, 이세계에 와서 처음으로 보상받는 순간. 전생에선 남의 회사 매출을 위해 똥을 치웠는데, 여기선 적어도 내가 쏠 마법을 위해 똥을 친다. 이거야말로 노동의 주체성 회복 아닌가.

에디터를 저장하고 닫는다.

```
:wq!
```

느낌표를 붙인 건, 리드 온리 경고를 강제로 짓밟고 저장하겠다는 뜻이다. 권한자만이 누릴 수 있는, 작지만 확실한 폭력. `[SYSTEM] ignite.mag saved.` 라는 짧은 로그가 한 줄 떴다가 사라졌다.

세계의 화염 마법이, 방금 막 버전 2로 패치됐다.

전 대륙의 마법사들은 아직 모른다. 그들이 수백 년간 3초씩 헌납해온 그 `sleep`이, 방금 한 몰락 남작가 폐급 도련님의 손끝에서 영원히 삭제됐다는 걸. 다음 패치 노트가 올라간다면 아마 이렇게 적힐 거다. **"화염 마법 캐스팅 속도가 개선되었습니다(영창 시간 3000ms → 0ms). 일부 마법사의 기존 워크플로우에 영향을 줄 수 있습니다."**

알 게 뭐야. 핫픽스는 빠를수록 좋은 거다.

나는 침대에서 천천히 몸을 일으켰다. 며칠을 앓아누웠던 탓에 다리가 후들거렸지만, 머릿속만은 야근 끝의 배포 직전처럼 또렷했다.

문제는, 코드를 짜는 것과 코드가 *도는 것*은 완전히 다른 얘기라는 거다.

이건 8년 차 짬밥이 뼛속에 새겨준 진리다. 로컬에서 멀쩡히 돌던 코드가 프로덕션에서 터지는 일은 일상다반사고, 리뷰에서 완벽해 보이던 PR이 머지하자마자 장애를 내는 것도 흔하다. 컴파일 됐다고 동작하는 게 아니고, 동작한다고 의도대로 도는 게 아니다.

방금 짠 `cast_ignite_v2`는 분명 깔끔하다. 논리적으로는 흠잡을 데가 없다. 마나 10, 즉시 가열, 점화. 머릿속 시뮬레이션으로는 백 번을 돌려도 완벽하다.

하지만 머릿속 시뮬레이션은 머릿속에서나 완벽한 법이다.

진짜 검증은 단 하나. 돌려보는 거다.

나는 방 한구석으로 시선을 던졌다. 낡은 철제 화로가 하나 놓여 있었다. 며칠째 불씨도 없이 식어 있는, 이 가난한 영주성의 초라한 난방기구.

테스트 타깃으로는 딱이었다.

물론, 마음 한구석이 살짝 근질거리긴 했다. `allocate_mana(10)`에서 `instant_ignition(1000)`. 1000도. 무영창. 비동기 즉시 발동. 이론상으로는 기존 파이어볼보다 한참 강해야 정상인데…… 얼마나 강할지는 아직 모른다. 변수를 안 잡은 채로 프로덕션에 쏘는 셈이다.

뭐, 어차피 테스트 환경 아닌가. 화로 하나 정도 녹는 거야 감수해야지. 스테이징 좀 깨먹는 게 개발자 일상이지.

나는 손가락을 풀었다. 우두둑.

입꼬리가 슬며시 올라갔다.

"자, 그럼."

허공의 터미널 창이, 내 명령을 기다리며 조용히 깜빡이고 있었다.

"테스트를 해볼까."
